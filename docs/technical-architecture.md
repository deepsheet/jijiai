## 1. 技术栈选型 (Tech Stack)

| 层级 | 技术选型 | 理由 | 成本 |
|------|----------|------|------|
| 应用框架 | Tauri v2 | 极小的包体积 (<5MB 核心)，低内存占用，Rust 后端安全性高，完美支持 Win/Mac 跨平台 | 免费 |
| 前端 UI | React + TypeScript + Tailwind | 开发效率高，生态丰富，易于实现现代化悬浮窗 UI | 免费 |
| 后端核心 | Rust | 高性能，直接绑定系统底层 API (音频、文件系统)，无 GIL 锁限制 | 免费 |
| 音频捕获 | cpal (Rust) | 跨平台音频抽象库，直接对接 Windows WASAPI / Mac CoreAudio | 免费 |
| ASR 引擎 | whisper.cpp (via llama-cpp-rs) | C++ 编写，Rust 绑定。支持 GPU 加速 (Metal/CUDA)，量化模型推理速度极快，适合本地运行 | 免费 |
| 流式处理 | aho-corasick (Rust) | 实时关键词匹配，毫秒级响应，用于第一层硬触发 | 免费 |
| 向量数据库 | LanceDB (Rust Native) | 基于磁盘的列式存储，内存占用极低，嵌入进程运行，无需独立服务，适合本地 RAG | 免费 |
| Embedding | BGE-M3 (ONNX Runtime) | 支持中文效果极佳，通过 ONNX Runtime 在 CPU/NPU 上高效推理 | 免费 |
| LLM 推理 | Qwen2.5-1.5B (GGUF) | 参数量小，速度快，中文指令遵循能力强，适合生成短话术 | 免费 |
| 云端备选 | LLM Adapter (OpenAI-compatible) | 支持 DeepSeek/通义千问/Ollama/GGUF 动态切换，成本约 0.02 元/分钟 | 按需付费 |

## 3. 核心难点与解决方案

### A. 跨平台音频捕获 (Virtual Audio Cable)

| 问题 | 解决方案 |
|------|----------|
| Win 和 Mac 的虚拟声卡驱动不同 | 安装包集成 VB-Cable Installer.exe (Win) 和 BlackHole.pkg (Mac)，首次启动时自动检测并引导安装 |
| 需要用户手动配置音频路由 | 前端提供"音频设置向导"页面，根据用户选择的场景模式，动态展示不同的配置步骤（特别是模式 A 的声音透传设置） |
| 驱动检测与枚举 | Rust 后端使用 cpal 枚举音频设备，判断 `VB-Audio Virtual Cable` 或 `BlackHole` 是否存在，若不存在则提示用户安装 |
| 管理员权限问题 | 安装驱动时请求提权，主程序无需管理员权限运行 |

### B. 场景化音频输入策略

| 问题 | 解决方案 |
|------|----------|
| 如何降低用户配置难度 | 提供三种预设场景模式（在线会议/电话辅助、线下面谈/个人演练、全双工纪要模式），用户只需选择场景，无需了解技术细节 |
| 如何避免回声问题 | 模式 C 强制要求佩戴耳机，UI 强提示；当虚拟声卡检测到语音时，暂时降低麦克风灵敏度或忽略麦克风输入 |
| 如何动态切换输入流 | Rust 音频模块支持运行时动态切换输入流（从单路切换到双路），无需重启应用；使用 cpal 的独立 Input Stream 管理 |
| 如何处理不同场景的性能需求 | 模式 A 优先保障延迟 < 0.8 秒；模式 B/C 延迟可放宽至 1.5 秒；若 CPU > 80%，自动降级（如模式 C 降级为模式 A） |

### C. 语义分析与意图识别 (三层动态触发机制)

| 问题 | 解决方案 |
|------|----------|
| 传统方案仅在句号/停顿触发，话术推荐延迟高 | 采用三层动态触发机制，从毫秒级到秒级分层处理，实现话术预热与实时推荐 |
| 碎片化识别导致话术逻辑断裂 | 通过 LLM 适配器进行全量校验，确保话术连贯性与准确性 |
| 供应商绑定导致架构僵化 | 设计 LLM Provider trait，支持云端/本地模型动态切换 |

#### 三层动态触发机制 (Three-Tier Stream Trigger)

| 层级 | 触发条件 | 实现方式 | 目标延迟 | 话术状态 |
|------|----------|----------|----------|----------|
| **第一层：本地流式硬触发 (Fast Path)** | ASR partial_result 中匹配强信号词 | Rust aho-corasick 实时扫描关键词（竞品、价格、否定词） | < 50ms | 预热 (Pre-warming) |
| **第二层：增量语义预测 (Medium Path)** | 单句长度 > 阈值 或 检测核心短语 | 提取当前片段进行本地向量检索 (LanceDB + BGE-M3) | 100-300ms | 候选 (Candidate) |
| **第三层：全量 LLM 校验 (Full Path)** | ASR 标点 或 VAD 检测 500ms 停顿 | LLM Adapter 调用云端/本地模型进行最终确认 | 500ms-2s | 确认 (Confirmed) |

**话术状态流转**：预热 → 候选 → 确认（仅向上升级，避免悬浮窗频繁跳变）

#### LLM 适配器设计 (LLM Adapter)

| 特性 | 实现方式 |
|------|----------|
| **解耦设计** | 后端封装统一 trait `LLMProvider`，严禁硬编码特定供应商 |
| **标准协议** | 采用 OpenAI-compatible API 标准，支持 `/chat/completions` 端点 |
| **动态切换** | 通过配置文件 `config.json` 动态指定供应商（deepseek / qwen / ollama / gguf） |
| **本地模型支持** | 支持 GGUF 格式模型（Qwen2.5、Llama3）通过 Ollama 或 llama.cpp 推理 |

**适配器接口定义 (Rust)**：
```rust
#[async_trait]
pub trait LLMProvider: Send + Sync {
    async fn chat(&self, messages: Vec<Message>) -> Result<String>;
    async fn health_check(&self) -> Result<bool>;
}
```

#### Rust 线程与并发优化

| 组件 | 线程模型 | 通信方式 | 说明 |
|------|----------|----------|------|
| ASR 线程 | 持续运行 | `mpsc::channel` | 实时推送 partial_result 与 final_result |
| 逻辑层线程 | 持续运行 | `mpsc::channel` + `Arc<Mutex>` | 并发执行关键词匹配、向量检索、LLM 调用 |
| UI 防抖模块 | 持续运行 | `tokio::time::sleep` | 话术状态变更后延迟 200ms 更新，避免频繁跳变 |

**并发流程**：
1. ASR 线程通过 `mpsc` 通道将流式文本推送到逻辑层
2. 逻辑层线程并发执行：
   - 第一层：aho-corasick 匹配强信号词 → 触发预热话术
   - 第二层：提取片段进行向量检索 → 返回候选话术
   - 第三层：积累完整句子后触发 LLM 校验 → 返回确认话术
3. UI 防抖模块接收话术状态变更，延迟更新悬浮窗

#### UI 防抖逻辑 (Pre-warming)

| 状态 | 显示行为 | 触发条件 |
|------|----------|----------|
| **预热 (Pre-warming)** | 悬浮窗淡入，灰色文字 | 第一层匹配到强信号词 |
| **候选 (Candidate)** | 悬浮窗高亮，蓝色文字 | 第二层向量检索返回结果 |
| **确认 (Confirmed)** | 悬浮窗固定，绿色文字 | 第三层 LLM 校验完成 |

**防抖策略**：
- 状态变更后延迟 200ms 更新 UI
- 若 200ms 内状态再次变更，重置计时器
- 确认态话术保留 3s 后自动消失

### D. 数据持久化与增量更新

| 问题 | 解决方案 |
|------|----------|
| 用户多次上传知识库，如何避免重复计算 | 对每个上传文件计算 MD5 哈希，LanceDB 中存储 file_hash 元数据 |
| 文件修改后的更新逻辑 | 先删除旧 Hash 对应的所有向量片段，再插入新片段 |
| 多用户数据隔离 | 每个用户的知识库存储在独立的 UUID 文件夹中 |
